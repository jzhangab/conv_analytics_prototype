LLM Classification

The Intent Classifier sends the user message (+ last 10 turns of history) to the LLM with a prompt listing all 10 intent definitions. The LLM returns a structured JSON with three fields:

{ "intent": "enrollment_forecasting",
  "confidence": 0.95,
  "reasoning": "User asks about forecasting enrollment..." }

Confidence Thresholds

0.85Main skill intents
0.75Data reasoning intent

Below the threshold, the intent is treated as None and routed to a fallback path. A lower threshold for data_reasoning allows softer follow-up questions to be recognized. Users can also type a number 1–7 to directly invoke a skill from the menu.

AI Reasoning & Planning

Deep research · iterative planning · multi-turn synthesis
Mode 1

Data Reasoning (Follow-up Analytics)

Triggered when the user asks a follow-up question about prior skill results. The system bridges structured data outputs into open-ended AI analysis through a user-confirmable planning loop.

1
Generate Analysis Plan
LLM formats all prior SkillResults (text + table summaries) as context and produces a concise 3-bullet plan describing exactly what analysis it will perform. Presented to user before any computation.
↓
2
User Reviews & Iterates on Plan
FSM enters ANALYSIS_PLANNING state. User can: yes → execute no → cancel or provide natural language feedback to revise the plan. LLM revises plan while keeping the 3-bullet structure. User can iterate multiple times.
↓
3
Execute & Synthesize
On confirmation, full results context (up to 30 rows per table, 3,000 chars per text) + web search context are sent to LLM with the confirmed plan → returns rich narrative analysis.
Mode 2

General Knowledge Search

Triggered when no skill intent matches and no prior results exist. The system conducts an agentic web-search loop to research the question before answering.

Output Guardrails

How the system ensures accurate, safe, and validated outputs
Phase 1 — Pre-Execution
Before any skill runs

Parameter Completeness Check

  • Schema defines all required vs optional params per skill
  • System checks for missing required params before proceeding
  • File uploads tracked separately via /upload endpoint
  • Missing params trigger a targeted user prompt — not a generic error

User Confirmation Gate

  • All parameters are shown to the user in a plain-language summary
  • Inherited params are flagged *(from prior request)*
  • User must explicitly confirm (yes) before skill executes
  • User can say edit + new value to revise any parameter
  • Parser handles synonyms: "yep / sure / ok" = yes, "change / modify" = edit

File Validation

  • Allowed extensions: csv, xlsx, xls, pdf, docx, txt
  • Max size: 25 MB
  • Protocol files: hard-capped at 800,000 characters (~200k tokens) before analysis
  • Site list files: column names inferred via LLM if non-standard headers
Phase 2 — During Execution
While the skill is running

Schema-Based Normalization

  • Choice params have canonical values + alias mappings
  • "Phase II" → "Phase 2", "lung cancer" → "NSCLC"
  • Country lists parsed from comma / semicolon / "and" delimited strings
  • Integer params validated as positive; dates normalized to YYYY-MM-DD

Deterministic Computation (Enrollment)

  • Stage 1: LLM estimates model parameters for 3 scenarios (pessimistic / moderate / optimistic)
  • Stage 2: Python computes all enrollment curves deterministically (logistic ramp + monthly math)
  • Stage 3: LLM narrates the pre-computed numbers
  • LLM cannot alter numerical outputs — auditable and reproducible

Jaro-Winkler Matching (Site Profiling)

  • Step 1: name + city match (threshold: JW > 0.90)
  • Step 2: first 3 address words + city (threshold: JW > 0.88)
  • Each CTMS site matches at most once — conflict resolved by best score
  • No LLM involved in matching — fully deterministic string algorithm
Phase 3 — LLM Output Validation
Structured response parsing and repair

JSON Response Validation

  • All LLM calls request strict JSON output (no markdown)
  • require_keys(data, keys, context) — raises ValueError with context if any expected key is absent
  • Enrollment: validates all 3 scenarios, all 3 sub-fields each
  • Reimbursement: validates overall_summary + country_assessments
  • safe_int() / safe_float() protect numeric fields with defaults

JSON Repair for Truncated Responses

  • LLM responses occasionally truncate at token limits mid-JSON
  • _repair_json() walks the partial string, tracking open brackets and quotes
  • Closes all unclosed strings, arrays, and objects
  • Allows downstream parsing to proceed with best-effort data rather than failing
  • Markdown code fences stripped before parsing

Web Search Graceful Degradation

  • All search calls wrapped in try/except
  • Failed or disabled search returns empty string silently
  • All agent prompts designed to work without web context
  • Never surfaces a search error to the user

AI Skills

What they are · how they work · how to add new ones

What Is a Skill?

A Skill (or SubAgent) is a domain-specific AI module that handles one analytical task end-to-end. It owns its own parameter schema, execution logic, LLM prompts, and output format. Skills are composable — results from one skill become context for the next.

Anatomy of a Skill
Intent ID
enrollment_forecasting
Unique key that the Intent Classifier maps user messages to
Parameter Schema
YAML-defined list of required + optional params with types, choices, aliases, and labels
Agent Class
EnrollmentForecastingAgent
Extends BaseAgent, implements run(params, state) → AgentResult
Prompt Templates
System + user message pairs in prompt_templates.py defining task, format rules, and output JSON spec
Router Registration
Entry in router.py registry maps intent ID → agent instance
AgentResult Output
text_response · table_data (list of dicts) · chart_json (Bokeh)

How to Add a New Skill

Adding a new skill requires 5 steps, touching 5 files. Each component is independent and well-scoped.

1
Add skill to config/skills_config.yaml
Define skill_id, display_name, description, required_parameters, and optional_parameters. Each param has a data_type (string / integer / choice / list / file / date) and optional choices with aliases.
2
Create backend/agents/my_skill_agent.py
Extend BaseAgent. Implement run(self, params: dict, state: ConversationState) → AgentResult. Use self.llm.complete_json() and self.web_search.search_for_skill() as needed. Return AgentResult(success, text_response, table_data, table_columns, chart_json).
3
Add intent to backend/orchestrator/intent_classifier.py
Add the new skill_id to VALID_INTENTS. Add keyword hints to the classification prompt so the LLM recognizes it. Add any legacy name aliases to _INTENT_ALIASES.
4
Register in backend/orchestrator/router.py
Add one line to the _registry dict: "my_skill": MySkillAgent(self.llm, config, self.web_search). The Orchestrator will call Router.get_agent("my_skill") when the intent is detected.
5
Add prompts to backend/llm/prompt_templates.py
Define MY_SKILL_SYSTEM and MY_SKILL_USER constants. The system prompt sets task scope, tone, and required JSON output schema. The user prompt injects parameters, web context, and history. Optionally add a skill-specific query builder in web_search.py.
💡
Optional: Add output validation in response_parser.py (require_keys) and a Bokeh chart builder in chart_builder.py if your skill produces visualizations.
